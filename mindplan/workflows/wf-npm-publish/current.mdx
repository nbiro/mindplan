# Publish to npm

## Purpose

Ship `mindplan-mcp` to the public npm registry so consumers can install without cloning from source. Own a **single reusable publish script** that maintainers run for the first claim and that `wf-npm-tag-publish` invokes non-interactively on version tags.

## Goal

A maintainer can run one scripted path; afterward any consumer can:

```bash
npm install -g mindplan-mcp
# or
npx mindplan-mcp init
```

and get a working CLI + MCP server whose `templates/` and `dist/` match this repo’s contract.

## Acceptance Criteria

- [ ] Publish script under the prescribed package encodes dry-run → test → publish → verify; CI can call non-interactive mode of the **same** script.
- [ ] Package name `mindplan-mcp` is claimed on the public registry at version `0.1.0` (or the next SemVer if already taken by us).
- [ ] Dry-run / pack listing asserts tarball contents: `dist/`, `templates/`, `README.md`, `SPEC.md`, `LICENSE` — not `src/`, tests, or local `mindplan/` territory.
- [ ] `npx mindplan-mcp --help` (or `mindplan-mcp help` after global install) shows init/view/help usage.
- [ ] README Quick start leads with npm / `npx`; clone+build remains a contributor path.
- [ ] `CONTRIBUTING.md` Publishing documents the script for first/manual publish and points later releases to `wf-npm-tag-publish`.

## Versioning policy

- **Scheme:** SemVer starting at current `package.json` version **`0.1.0`**.
- **Until 1.0.0:** treat the public API as evolving; prefer `minor` for additive changes, `patch` for fixes; document breakage in changelog/SPEC when not bumping major on 0.x.
- **First publish:** publish whatever is currently in `package.json` (expected `0.1.0`).
- **Later releases:** bump version on **main**, push matching `vX.Y.Z` tag; automation is `wf-npm-tag-publish`.

## Release style (this Workflow)

**Script-first, manual first claim.** Maintainer machine + npm account with publish rights for the initial `npm publish` (OTP / `npm login` as required).

Depends on `f-npm-registry` for **package identity + OIDC contract docs** — not for a live Trusted Publisher binding (that binding is `wf-npm-tag-publish` after this Workflow claims the package).

Handoff: once the script works and the package is claimed, live Trusted Publishing + tag automation is `wf-npm-tag-publish` (using conventions from `f-npm-registry` / `f-github-actions`).

Out of scope for this node:

- `.github/workflows/publish.yml` (owned by `wf-npm-tag-publish`)
- Live npm Trusted Publishing UI binding
- Automated changelog generation

## Execution Logic

Package: `src/workflows/wf-npm-publish/`. Depends on: `f-mcp-runtime`, `f-npm-registry`. Belongs to: `j-npm-publish`, `j-agent-onboarding`.

### Publish script location

- **Primary:** implement the script inside `src/workflows/wf-npm-publish/` (e.g. `publish.mjs` or `publish.sh`).
- **Optional thin wrapper:** a root `package.json` script (e.g. `"publish:release": "node src/workflows/wf-npm-publish/publish.mjs"`) so maintainers and CI share one entrypoint.
- Do **not** invent a parallel root-only `scripts/publish.sh` that diverges from the package implementation.

### Publish script (source of truth)

1. **Preflight:** assert clean git tree (or documented allowlist); confirm `package.json` name/version/bin/files/engines/repository URLs.
2. **Dry-run:** `npm pack --dry-run` / `npm publish --dry-run` — assert allowlisted tarball paths; no secrets, no `.cursor/`, no local `mindplan/` territory, no `src/`.
3. **Test:** `npm test` green (required in both maintainer and CI modes).
4. **Publish:** `npm publish` (`prepublishOnly` builds `dist/`). Interactive login/OTP only in maintainer mode; CI mode skips login and relies on OIDC.
5. **Verify:** fresh install — `mindplan-mcp help` + `mindplan-mcp init` in a temp directory (maintainer mode; CI may use a lighter post-publish check).

### First publish runbook

1. Confirm registry identity aligns with github.com/nbiro/mindplan (`f-npm-registry` docs).
2. Run the publish script in maintainer mode on the release commit (claims the name).
3. Sync docs (README + CONTRIBUTING).
4. Do **not** block on Trusted Publisher UI — that is `wf-npm-tag-publish` after claim.

### Docs sync

- README Quick start: npm / `npx` primary.
- CONTRIBUTING: first publish = run the script after `npm login`; later = bump on main, tag, push (automation lands with `wf-npm-tag-publish`).

## Failure / recovery

- **Name taken by someone else:** stop; escalate — do not rename without a product decision.
- **`0.1.0` already published by us:** bump patch/minor and publish; update docs.
- **Broken publish after install check:** ship a patch via the same script; never leave README advertising a broken path.
