# Init a consumer project

## Purpose

Scaffold a consumer repo for MindPlan: empty `mindplan/` territory layout, project config (`mindplan/config.json`) choosing prescribed vs layout-free packages, always-on agent playbook, define-entities / plan-project / review-work / code-review skills, MCP example config, per-agent integration guides, root `AGENTS.md`, Cursor-native copies under `.cursor/skills/` and `.cursor/rules/mindplan.mdc`, `.cursorignore` that ignores the derived map (and agent asset copies under `mindplan/agent/**`) while allowing agent file tools to edit territory MDX prose, and `.cursor/permissions.json` that allowlists MindPlan MCP tools so Cursor Auto-review does not prompt on playbook-required graph mutations.

When MindPlan itself is upgraded, consumers re-run `mindplan-mcp init -f` / `--force` to reinstall the latest bundled agent assets over existing copies (playbook, skills, Cursor copies, integrations, etc.). Bare `init` remains idempotent (skip if dest exists). Project config (`mindplan/config.json`) is **not** overwritten by `-f` alone — only by an explicit `--layout` (preserves a layout-free project's choice on upgrade).

## Execution Logic

Package: `src/workflows/wf-project-init/`

- `init.ts` — `initProject` plus install helpers (exported for CLI wiring in `f-mcp-runtime`):
  - Shared option: `{ force?: boolean }` on every agent-asset installer. When `force` is false/omitted: skip if dest exists (`installed: false`). When `force` is true: overwrite files; for directories, replace the dest tree then copy from template (`installed: true`).
  - `installProjectConfig` — writes `mindplan/config.json` with `implementation_packages`: `required` (default / `--layout prescribed`) or `off` (`--layout free`). Missing config is treated as `required` by loaders elsewhere. Idempotent: create when missing; update only when an explicit `--layout` flag is passed. **Not** controlled by agent-asset `-f` — layout force stays `--layout`-only.
  - `installAgentPlaybook` — `templates/agent/playbook.md` → `mindplan/agent/`
  - `installDefineEntitiesSkill` — skill under `mindplan/agent/skills/define-entities/`
  - `installPlanProjectSkill` — skill under `mindplan/agent/skills/plan-project/`
  - `installReviewWorkSkill` — skill under `mindplan/agent/skills/review-work/`
  - `installCodeReviewSkill` — thin skill under `mindplan/agent/skills/code-review/`
  - `installCursorSkills` — copy skills into Cursor discovery paths from package templates:
    - → `.cursor/skills/mindplan-define-entities/`
    - → `.cursor/skills/mindplan-plan-project/`
    - → `.cursor/skills/mindplan-review-work/`
    - → `.cursor/skills/mindplan-code-review/`
    Returns one install result per skill path so the CLI can report each path. Source is `templates/agent/skills/…`, not the on-disk `mindplan/agent` copy. Honors `{ force }`.
  - `installCursorRule` — write `.cursor/rules/mindplan.mdc`: Cursor rule frontmatter (`alwaysApply: true`) plus the playbook body from `templates/agent/playbook.md`. Skip if present unless `force`.
  - `installMcpExample` — `mcp.json.example`
  - `installAgentIntegrations` — guides into `mindplan/agent/integrations/` (Cursor guide documents that init installs `.cursor/skills` + `.cursor/rules`; manual copy only if those files were deleted without `-f`)
  - `installRootAgentsMd` — root `AGENTS.md` from template
  - `installCursorIgnore` — installs `templates/agent/cursorignore` (ignores `mindplan/map.md` + `mindplan/agent/**`; does **not** block `current.mdx` / `next.mdx`). Comment must match real behavior: init copies agent instructions to `.cursor/rules` and `.cursor/skills`
  - `installCursorPermissions` — installs `templates/agent/permissions.json` → `.cursor/permissions.json`

CLI call/report of these helpers (including parsing `-f` / `--force` and passing `{ force: true }`) lives in `f-mcp-runtime` (paired `next`). Smoke asserts for force overwrite live in `wf-test-harness` (paired `next`).

Depends on: `f-territory-store`. Belongs to: `j-agent-onboarding`. Consumer use case: `wf-layout-free`.

## Acceptance Criteria

- `mindplan-mcp init` creates consumer mindplan + agent assets idempotently (skip existing dests) when `-f` is absent
- With `{ force: true }`, every agent-asset installer overwrites existing files/dirs from templates and returns `installed: true`
- `init` writes `mindplan/config.json`; `--layout free` sets `implementation_packages` to `off`; default/prescribed sets `required`; bare `-f` does **not** change an existing config's layout mode
- Playbook, define-entities, plan-project, review-work, and thin code-review skills, integrations, MCP example, AGENTS.md, `.cursorignore`, and `.cursor/permissions.json` are installed from `templates/`
- This package exports `installCodeReviewSkill`, `installCursorSkills`, and `installCursorRule` (and the other installers) that accept optional `{ force?: boolean }`
- Template `.cursorignore` does not list `mindplan/**/current.mdx` or `mindplan/**/next.mdx`, and its comment accurately describes the `.cursor/rules` + `.cursor/skills` copies
- Cursor integration guide states that init installs those Cursor paths (manual copy only if missing; or re-run `init -f` after upgrade)
- Template `.cursor/permissions.json` allowlists `mindplan:*` and steers Auto-review for playbook mutations
- Init helpers live in this package (not in the territory store)
- CLI wiring/reporting is owned by `f-mcp-runtime` next; force-overwrite smoke asserts by `wf-test-harness` next
