# Framework documentation

## Purpose

Own MindPlan’s normative documentation surfaces so agents and humans share one process: taxonomy, compiler gates, Territory Completeness, Foundation roles, plan-only planning sessions, territory authority split (MCP = graph; agent file tools = prose), layout modes (prescribed packages vs layout-free brownfield adoption), and how to evolve shipped work without destroying the living contract.

## Execution Logic

Docs live at repo root and under `templates/agent/` / `mindplan/agent/`. The package folder `src/workflows/wf-framework-docs/` is a pointer (no runtime TS).

Surfaces owned by this Workflow:

- `SPEC.md` — normative language (taxonomy, versioning, Territory Completeness, compiler rules, authority split, mutate `changed_files`, **`mindplan/config.json` / `implementation_packages`**)
- `AGENTS.md` — always-applied agent playbook (repo root)
- `templates/agent/playbook.md` — consumer playbook shipped by `mindplan-mcp init`
- `mindplan/agent/playbook.md` — local mirror of the playbook
- `templates/agent/skills/define-entities/` (+ mindplan/agent mirror) — entity scaffolding skill (note when packages are not scaffolded)
- `templates/agent/skills/plan-project/` (+ mindplan/agent mirror) — plan-only sessions (no application code)
- `README.md` — product framing, taxonomy summary, and brownfield layout-free adoption

Foundation scaffold copy that mentions roles lives in `src/foundations/f-territory-store/store.ts` (touched when role/description conventions change).

### Layout modes

| Mode | Config | When |
|------|--------|------|
| Prescribed (default) | `implementation_packages: "required"` or missing file | Greenfield / this MindPlan repo — architecture screams from `src/foundations|workflows/<id>/` |
| Layout-free | `implementation_packages: "off"` | Brownfield — graph + territory + playbook only; agents implement in the existing app layout |

Set at init via `mindplan-mcp init --layout free|prescribed` (`wf-project-init` / `wf-layout-free`). Compiler graph gates are identical; only package scaffolding and package/dirty-src integrity checks differ.

### Territory Completeness

`current.mdx` is the complete contract of what a node *is* in the repository now. `next.mdx` is a complete proposed successor of that same contract (seeded from current, edited in place). Atomic Ops on `next` may be evolution-scoped; PRD/spec sections must remain a full post-ship document. Delta-only or changelog-only `next` bodies are forbidden because `ship` promotes `next` over `current`.

### Foundation roles (docs convention)

Assembler, Infra, Design system, Adapter — documentation groupings only (not NodeTypes). Agents SHOULD tag the role at the start of a Foundation `description`.

### Plan-only sessions

When the user wants to model or restructure the product plan without writing application code, agents follow `mindplan/agent/skills/plan-project/`. The always-on playbook routes those sessions to that skill and forbids advancing into implementation (`in-progress` / `src/`) until a later execution session.

### Authority split (graph vs prose)

- **MCP only:** `create_node`, `link_nodes`, `unlink_nodes`, `update_node_status`, `open_next`, `discard_next`, and all server-owned frontmatter (`state`, timestamps, edge arrays).
- **Agent file tools (preferred):** `title`, `description`, body (PRD / Atomic Ops), checkbox toggles at `current_path` / `next_path` from orientation — so host “changed files” UIs show prose edits.
- **`patch_node_territory`:** optional fallback for automation / weak file-tool hosts.
- **Orient** via MCP; never treat on-disk frontmatter as graph authority — trust MCP `record`.
- Keep `mindplan/map.md` out of agent ignore-bypass as a derived snapshot only.

## Acceptance Criteria

- SPEC documents Territory Completeness, Foundation roles, the MCP=graph / file-tools=prose authority split, and `implementation_packages` required vs off
- Playbooks (AGENTS + templates + mindplan mirror) instruct agents for both layout modes and forbid hand-editing server-owned frontmatter
- Playbooks route plan-only / architecture-modeling asks to the `plan-project` skill
- define-entities and plan-project skills align with the authority split and layout-free (no forced package paths when `off`)
- README summarizes Foundation roles, agent skills, and brownfield layout-free adoption
- Active MindPlan Journey/Foundation/Workflow territories describe full repo state (not stubs or latest-change deltas)
