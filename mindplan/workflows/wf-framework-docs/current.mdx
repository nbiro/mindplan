# Framework documentation

## Purpose

Own MindPlan’s normative documentation surfaces so agents and humans share one process: taxonomy, compiler gates, Territory Completeness, Foundation roles, plan-only planning sessions, orchestrated Plan Review and Implementation review loops (independent Reviewer subagent; findings via agent messages — not territory Review Notes), territory authority split (MCP = graph; agent file tools = prose), layout modes (prescribed packages vs layout-free brownfield adoption), Git delivery (feature-branch startup before any edits), and how to evolve shipped work without destroying the living contract.

## Execution Logic

Docs live at repo root and under `templates/agent/` / `mindplan/agent/`. The package folder `src/workflows/wf-framework-docs/` is a pointer (no runtime TS).

Surfaces owned by this Workflow:

- `SPEC.md` — normative language (taxonomy, versioning, Territory Completeness, compiler rules, authority split, mutate `changed_files`, **`mindplan/config.json` / `implementation_packages`**)
- `AGENTS.md` — always-applied agent playbook (repo root)
- `templates/agent/playbook.md` — consumer playbook shipped by `mindplan-mcp init`
- `mindplan/agent/playbook.md` — local mirror of the playbook
- `templates/agent/skills/define-entities/` (+ mindplan/agent mirror) — entity scaffolding skill (note when packages are not scaffolded)
- `templates/agent/skills/plan-project/` (+ mindplan/agent mirror) — plan-only sessions (no application code); run Plan Review loop to `ready`
- `templates/agent/skills/review-work/` (+ mindplan/agent mirror) — Plan Review (`draft → ready`) and Implementation review (`in-review → ship` / `resolved`) as Reviewer role for a spawned subagent
- `templates/agent/skills/code-review/` (+ mindplan/agent mirror) — thin portable Implementation code-review checklist; prefer host-native built-in skill/command (e.g. Cursor `/code-review`), else community `code-review-skill`, else this thin skill
- `README.md` — product framing, taxonomy summary, and brownfield layout-free adoption

Foundation scaffold copy that mentions roles lives in `src/foundations/f-territory-store/store.ts` (touched when role/description conventions change).

### Layout modes

| Mode | Config | When |
|------|--------|------|
| Prescribed (default) | `implementation_packages: "required"` or missing file | Greenfield / this MindPlan repo — architecture screams from `src/foundations|workflows/<id>/` |
| Layout-free | `implementation_packages: "off"` | Brownfield — graph + territory + playbook only; agents implement in the existing app layout |

Set at init via `mindplan-mcp init --layout free|prescribed` (`wf-project-init` / `wf-layout-free`). Compiler graph gates are identical; only package scaffolding and package/dirty-src integrity checks differ.

### Territory Completeness

`current.mdx` is the complete contract of what a node *is* in the repository now. `next.mdx` is a complete proposed successor of that same contract (seeded from current, edited in place). Atomic Ops on `next` may be evolution-scoped; PRD/spec sections must remain a full post-ship document. Delta-only or changelog-only `next` bodies are forbidden because `ship` promotes `next` over `current`.

### Foundation roles (docs convention)

Assembler, Infra, Design system, Adapter — documentation groupings only (not NodeTypes). Agents SHOULD tag the role at the start of a Foundation `description`.

### Plan-only sessions

When the user wants to model or restructure the product plan without writing application code, agents follow `mindplan/agent/skills/plan-project/`. Planning sessions finish modeled Foundations/Workflows at `draft` (links, PRD, Atomic Ops complete), then run the **Plan Review loop** (spawn independent Reviewer) until `ready` or escalate — they do not self-advance to `ready`. After `ready`, plan-only sessions stop (no `in-progress` / `src/`). **“Ship the plan”** means finish at `draft` and run that Plan Review loop to `ready` — not build-pipeline `ship`.

### Review gates (orchestrated loop)

Skill: `mindplan/agent/skills/review-work/`. Parent (author/implementer) **orchestrates**; does not self-approve.

1. **Plan Review** — `draft → ready` for Workflows/Foundations. Checks buildability, domain fit, dependency completeness, decomposition quality, and scope before any code exists.
2. **Implementation review** — `in-review → ship` / Bug `resolved`. Evidence-checks every Atomic Op; verifies fit (domain, dependency accuracy, decomposition) and territory-vs-diff drift.

**Loop (both gates):**

1. Parent reaches the gate (`draft` complete, or all Atomic Ops `[x]` then `update_node_status → in-review`).
2. Parent spawns a **fresh** independent Reviewer subagent (Cursor `Task` or host equivalent) each round — never `resume` a prior Reviewer; never same-session self-approve. Prefer synchronous; if background-forced, wait for completion before advancing.
3. Reviewer follows `review-work`, calls `update_node_status` only on verdict, returns a **structured verdict message** to the parent. Reviewer **must not** write `## Review Notes` (or equivalent) into `current.mdx` / `next.mdx`.
4. Parent re-reads MCP (`record.state` / `next.state`). Chat claim alone is insufficient.
5. Approve + MCP confirms → continue. Reject + MCP confirms → parent fixes from Findings in the message, re-enters the gate (Implementation: re-check Atomic Ops and advance to `in-review` again before re-spawn), re-spawns.
6. Approve claimed but state unchanged (Blocked) → failed transition; re-spawn after fixing the block, or escalate. Counts toward the failure budget.
7. After **3** rejects/failed transitions at the same gate → escalate to the human with the latest verdict message. Do not self-approve.

**Structured verdict message** (Reviewer → Parent): `Verdict` (Approve|Reject), `Procedure` (PlanReview|ImplementationReview), `Node` (+ slot when evolving), `Findings` (itemized; Evidence+Fit on Implementation Approve), `StatusAttempted`.

### Implementation code review (host-native first)

Procedure B **also** runs a general code review, using the **host agent’s** capability when available:

1. **Host built-in** — If the host exposes a native code-review skill or command (e.g. Cursor `/code-review`, Claude Code bundled `/code-review`, Cursor `review` / `review-bugbot` when that is the host’s standard), follow its instructions. Load the matching `SKILL.md` from the host skill dirs when one exists. Slash commands cannot always be invoked from a subagent UI — apply the command’s published criteria (bugs, regressions, security, missing tests; findings primary; no drive-by edits) as if that command were run.
2. **Else community skill** — If `code-review-skill` or `code-review` is installed in host skill dirs (e.g. [awesome-skills/code-review-skill](https://github.com/awesome-skills/code-review-skill)), follow that `SKILL.md` (load only language guides that match the diff).
3. **Else thin MindPlan skill** — Follow `mindplan/agent/skills/code-review/SKILL.md` (installed by init).
4. Fold **blocking** findings into the structured verdict `Findings`. Blocking → Reject (do not ship).
5. Init does **not** vendor open-source trees — only the thin skill is guaranteed.

Parent spawn prompts SHOULD name the host affordance when known (e.g. “also apply Cursor `/code-review` criteria”).

**Diff hygiene:** Reject if the working tree / branch diff includes scratch helpers, one-off patch scripts (e.g. `_patch_*.py`), or other files outside the node’s Atomic Ops / package ownership. Implementer must delete them before re-review.

Hosts without subagents: independent Reviewer chat (legacy). Still preferred over self-approve.

Reviewer inherits the parent’s git branch; does not re-run the main→feature-branch ritual mid-loop.

### Authority split (graph vs prose)

- **MCP only:** `create_node`, `link_nodes`, `unlink_nodes`, `update_node_status`, `open_next`, `discard_next`, and all server-owned frontmatter (`state`, timestamps, edge arrays).
- **Agent file tools (preferred):** `title`, `description`, body (PRD / Atomic Ops), checkbox toggles at `current_path` / `next_path` from orientation — so host “changed files” UIs show prose edits.
- **`patch_node_territory`:** optional fallback for automation / weak file-tool hosts.
- **Orient** via MCP; never treat on-disk frontmatter as graph authority — trust MCP `record`.
- Keep `mindplan/map.md` out of agent ignore-bypass as a derived snapshot only.

### Git delivery (feature-branch startup)

Playbooks (`AGENTS.md`, `templates/agent/playbook.md`, `mindplan/agent/playbook.md`) MUST instruct agents to leave `main`/`master` **before any repo writes**, not only before commit. Applies to plan-only, Reviewer, and implementation sessions alike.

**Repo writes** include host file-tool edits **and** MCP graph mutations that write files (`create_node`, `open_next`, `link_nodes`, `update_node_status`, etc. via `changed_files`). Normative text lives only in the playbooks; integration guides (`wf-agent-integrations`) may point at this section and must not duplicate it.

**When already on the correct feature branch** for this session’s owning work: do **not** re-run the startup ritual — continue on that branch. Do not checkout `main` mid-session if it would abandon or conflict with in-flight WIP. Re-run only when starting work that should **not** land on the current branch (new owning node id, or the user asked for a fresh branch from latest main).

**When HEAD is on `main`/`master` (or detached), or starting work that should not land on the current branch:**

1. If the working tree is dirty in a way that blocks checkout/pull: **stop** — ask the user how to proceed (stash, commit on a branch, or discard). Do not invent destructive resets.
2. `git checkout main` (or `master`)
3. `git pull --ff-only` — if this fails (diverged history, conflicts): **stop** and report; do not create merge commits on main
4. Create or switch to a **feature branch** (prefer naming from the owning node id, e.g. `wf-integrity-check`)
5. **Only then** perform repo writes (file tools or MCP mutations)

Rationale: reduces the risk that concurrent agent sessions both write on `main`. Landing remains feature branch + PR; never push to `main`/`master`.

## Acceptance Criteria

- SPEC documents Territory Completeness, Foundation roles, the MCP=graph / file-tools=prose authority split, and `implementation_packages` required vs off
- Playbooks (AGENTS + templates + mindplan mirror) instruct agents for both layout modes and forbid hand-editing server-owned frontmatter
- Playbooks route plan-only / architecture-modeling asks to the `plan-project` skill; **“Ship the plan”** means finish at `draft` and run the Plan Review loop until `ready` (or escalate) — not park for a human, and not build-pipeline `ship`
- Playbooks define the **Review loop** for Plan Review and Implementation review: spawn independent Reviewer subagent, structured verdict message (no territory Review Notes), MCP re-read, re-entry rules, failed-transition handling, 3-reject escalate
- Playbooks **Git delivery** instructs: when on `main`/`master` or starting fresh work, checkout default branch → `pull --ff-only` → feature branch → **then** any repo write (file tools or MCP); skip ritual when already on the correct feature branch; stop on dirty-tree or non-ff-only pull failure; Never-do forbids writing on `main`/`master`
- define-entities, plan-project, review-work, and thin code-review skills align with the Review loop; Implementation review prefers **host-native** code-review (built-in skill/command) then community `code-review-skill`, else thin skill; Reject on scratch/unrelated files; findings via agent message
- README and SPEC abstract summarize Foundation roles, agent skills (including `review-work`), and brownfield layout-free adoption
- `mindplan-mcp init` + integration guides install/discover `review-work` (owned with `wf-project-init` / `wf-agent-integrations` / `f-mcp-runtime`); guides keep pointing at playbook Git delivery (no duplicate normative ritual in this Workflow). Integration one-liner wording may lag until a separate `wf-agent-integrations` evolution
- Active MindPlan Journey/Foundation/Workflow territories describe full repo state (not stubs or latest-change deltas)
