# Framework documentation

## Purpose

Own MindPlan‚Äôs normative documentation surfaces so agents and humans share one process: taxonomy, compiler gates, Territory Completeness, Foundation roles, plan-only planning sessions, Plan Review and Implementation review handoffs, territory authority split (MCP = graph; agent file tools = prose), layout modes (prescribed packages vs layout-free brownfield adoption), and how to evolve shipped work without destroying the living contract.

## Execution Logic

Docs live at repo root and under `templates/agent/` / `mindplan/agent/`. The package folder `src/workflows/wf-framework-docs/` is a pointer (no runtime TS).

Surfaces owned by this Workflow:

- `SPEC.md` ‚Äî normative language (taxonomy, versioning, Territory Completeness, compiler rules, authority split, mutate `changed_files`, **`mindplan/config.json` / `implementation_packages`**)
- `AGENTS.md` ‚Äî always-applied agent playbook (repo root)
- `templates/agent/playbook.md` ‚Äî consumer playbook shipped by `mindplan-mcp init`
- `mindplan/agent/playbook.md` ‚Äî local mirror of the playbook
- `templates/agent/skills/define-entities/` (+ mindplan/agent mirror) ‚Äî entity scaffolding skill (note when packages are not scaffolded)
- `templates/agent/skills/plan-project/` (+ mindplan/agent mirror) ‚Äî plan-only sessions (no application code); parks at `draft` for Plan Review
- `templates/agent/skills/review-work/` (+ mindplan/agent mirror) ‚Äî Plan Review (`draft ‚Üí ready`) and Implementation review (`in-review ‚Üí ship` / `resolved`)
- `README.md` ‚Äî product framing, taxonomy summary, and brownfield layout-free adoption

Foundation scaffold copy that mentions roles lives in `src/foundations/f-territory-store/store.ts` (touched when role/description conventions change).

### Layout modes

| Mode | Config | When |
|------|--------|------|
| Prescribed (default) | `implementation_packages: "required"` or missing file | Greenfield / this MindPlan repo ‚Äî architecture screams from `src/foundations|workflows/<id>/` |
| Layout-free | `implementation_packages: "off"` | Brownfield ‚Äî graph + territory + playbook only; agents implement in the existing app layout |

Set at init via `mindplan-mcp init --layout free|prescribed` (`wf-project-init` / `wf-layout-free`). Compiler graph gates are identical; only package scaffolding and package/dirty-src integrity checks differ.

### Territory Completeness

`current.mdx` is the complete contract of what a node *is* in the repository now. `next.mdx` is a complete proposed successor of that same contract (seeded from current, edited in place). Atomic Ops on `next` may be evolution-scoped; PRD/spec sections must remain a full post-ship document. Delta-only or changelog-only `next` bodies are forbidden because `ship` promotes `next` over `current`.

### Foundation roles (docs convention)

Assembler, Infra, Design system, Adapter ‚Äî documentation groupings only (not NodeTypes). Agents SHOULD tag the role at the start of a Foundation `description`.

### Plan-only sessions

When the user wants to model or restructure the product plan without writing application code, agents follow `mindplan/agent/skills/plan-project/`. Planning sessions leave modeled Foundations/Workflows at `draft` (links, PRD, Atomic Ops complete) and hand off for Plan Review ‚Äî they do not self-advance to `ready`. The always-on playbook forbids advancing into implementation (`in-progress` / `src/`) until a later execution session after Plan Review has moved the node to `ready`.

### Review gates

Two independent Reviewer sessions (skill: `mindplan/agent/skills/review-work/`):

1. **Plan Review** ‚Äî `draft ‚Üí ready` for Workflows/Foundations. Checks buildability, domain fit, dependency completeness, decomposition quality, and scope before any code exists. Reviewer calls `update_node_status` only; writes `## Review Notes` into the body.
2. **Implementation review** ‚Äî `in-review ‚Üí ship` / Bug `resolved`. Evidence-checks every Atomic Op; verifies fit (domain, dependency accuracy, decomposition) and territory-vs-diff drift. Same mutation boundary.

Plan authors and implementers park and hand off; they do not self-advance through these gates.

### Authority split (graph vs prose)

- **MCP only:** `create_node`, `link_nodes`, `unlink_nodes`, `update_node_status`, `open_next`, `discard_next`, and all server-owned frontmatter (`state`, timestamps, edge arrays).
- **Agent file tools (preferred):** `title`, `description`, body (PRD / Atomic Ops), checkbox toggles at `current_path` / `next_path` from orientation ‚Äî so host ‚Äúchanged files‚Äù UIs show prose edits.
- **`patch_node_territory`:** optional fallback for automation / weak file-tool hosts.
- **Orient** via MCP; never treat on-disk frontmatter as graph authority ‚Äî trust MCP `record`.
- Keep `mindplan/map.md` out of agent ignore-bypass as a derived snapshot only.
