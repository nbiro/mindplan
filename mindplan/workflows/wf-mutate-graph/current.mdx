# Mutate graph

## Purpose

Write-path MindPlan tools: create/link nodes, advance pipeline state, optional territory patches, and open/discard/promote evolutions — always through the compiler. Successful graph mutations return `changed_files` (repo-relative paths written or deleted) so hosts can surface what MCP touched.

## Execution Logic

Package: `src/workflows/wf-mutate-graph/`

- `tools.ts` — ownership list `MUTATE_TOOLS` registered by `f-mcp-runtime/server.ts`:
  - `create_node`, `link_nodes`, `unlink_nodes`
  - `update_node_status` (current or `next` slot; `ship` promotes `next`)
  - `patch_node_territory` (optional fallback for body/title/description/checkbox toggles; defaults to `next` when evolving)
  - `open_next`, `discard_next`

Interactive agents prefer host file tools for territory prose (`title` / `description` / body / checkboxes) at `current_path` / `next_path`; MCP remains mandatory for graph structure and server-owned frontmatter. Graph mutate responses include `changed_files`. `patch_node_territory` remains for automation and hosts with weak file tools.

Compiler gates live in `f-compiler-rules`. Depends on: `f-mcp-runtime`, `f-compiler-rules`. Belongs to: `j-territory-sdlc`.

## Acceptance Criteria

- All seven mutate tools are registered and persist through the territory store
- Illegal mutations return `Blocked:` and leave the graph unchanged
- Successful graph mutations refresh `mindplan/map.md` via the runtime
- Successful graph mutations (`create_node`, `link_nodes`, `unlink_nodes`, `update_node_status`, `open_next`, `discard_next`) return `changed_files` listing territory paths, optional `src/` scaffolds, and `mindplan/map.md` when refreshed
- `patch_node_territory` remains usable and returns `changed_files` / `path` for the slot written
- Verified via `npm test`
